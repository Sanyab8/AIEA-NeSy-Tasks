# Task 9: LangGraph Sports Knowledge Base Inference Engine

For Task 9, I migrated my Task 8 LangChain logical inference engine into LangGraph. My Task 8 solution used `ChatOpenAI` and a LangChain prompt to translate English questions into Prolog queries. It then used Janus SWI-Prolog to execute those queries against my Task 4 sports knowledge base and printed a trace explaining the result. In Task 9, I kept the same basic logic, but reorganized the system into a LangGraph workflow so that each step of the reasoning process is separated into its own node.

I first read the LangGraph overview and “Why LangGraph” documentation. LangGraph is useful for this project because it lets a program run as a stateful graph instead of one long chain. For this implementation, I used LangGraph’s `StateGraph` to build a workflow with nodes for retrieving context, judging relevance, refining context, generating a Prolog query, executing the query, and making a trace. This made the inference process easier to inspect because each node updates the shared state before passing it to the next step.

I used my Task 4 sports Prolog knowledge base. The KB contains students, sports, facts about who plays which sport, facts about indoor sports, facts about team sports, and rules such as `plays_indoor(X)`, `plays_team_sport(X)`, `busy_player(X)`, `student_athlete(X)`, and `plays_outdoor(X)`. This gives the project a small but useful knowledge base with around 10–20 facts and 5 rules.

The LangGraph workflow starts with a RAG-style retrieval step. The `retrieve_context` node reads the Prolog knowledge base and pulls lines that appear relevant to the English question. For example, if the question asks whether Sanya plays an indoor sport, the retrieval step pulls facts about Sanya, basketball, and indoor sports. Then the `judge_relevance` node uses the LLM to decide whether the retrieved context is relevant enough to answer the question. If the context is not relevant, LangGraph routes the workflow to the `refine_context` node, which retrieves the full knowledge base as an additional retrieval/refinement step. This satisfies the refinement requirement without relying on hidden chain-of-thought.

After the context is ready, the `generate_query` node uses the same LangChain-style prompt from Task 8 to translate the English question into a Prolog query. For example, “Does Sanya play an indoor sport?” becomes `plays_indoor(sanya)`, and “Is Bob a busy player?” becomes `busy_player(bob)`. The `ask_prolog` node then sends the generated query to Janus SWI-Prolog. Finally, the `make_trace` node creates a trace showing the query, the supporting facts/rules, and the final true/false result.

I tested the project using four questions: “Does Sanya play an indoor sport?”, “Does Bob play an indoor sport?”, “Is Bob a busy player?”, and “Is Sam a student athlete?” The first test returned true because Sanya plays basketball and basketball is listed as indoor. The second test returned false because Bob plays soccer and baseball, and neither is listed as an indoor sport. The third test returned true because Bob plays two different sports, so he satisfies the `busy_player(X)` rule. The fourth test was intended to test `student_athlete(sam)`, which should be true if Sam is a student and plays a sport. During testing, I found that Prolog raised an error because `student_athlete/1` was missing from the KB, so I added that rule to complete the knowledge base.

Overall, this Task 9 version successfully migrates the Task 8 LangChain solution into LangGraph. The project now uses a graph-based workflow, RAG-style context retrieval, AI-based relevance judgment, additional retrieval for refinement, LLM-based Prolog query generation, Prolog execution, and trace output. This makes the system more modular and easier to debug than the original Task 8 chain.
